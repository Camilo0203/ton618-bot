# TON618 Distribution Strategy

Commercial launch strategy for TON618 V1.

## Goal

Launch TON618 as an English-first Discord support product with:

- Free for core ticketing
- Pro for advanced operations
- Supporter for recognition-only donations

## Audience

- Discord communities with active staff teams
- Gaming communities with structured support needs
- SaaS or product communities using Discord for customer support
- Creator or Web3 communities that need staff workflows, not just a basic ticket panel

## Core message

TON618 is not a generic Discord bot. It is an ops console for staff teams that need tickets, SLA visibility, incident workflows, and live playbooks inside Discord.

## Distribution pillars

### 1. Product-led proof

- Use TON618 in your own support server first
- Capture screenshots of `/config tickets`, `/stats sla`, and `/ticket playbook`
- Collect 2-3 real before/after stories from servers using it

### 2. English-first content

- README and Quick Start should point to English-first commands
- Demo videos and screenshots should use English guild copy
- Community posts should describe the Free + Pro model clearly

### 3. High-intent communities

Focus first on places where owners or support leads already feel the problem:

- Discord server owner communities
- Discord bot communities
- self-hosting communities
- creator support communities
- gaming staff and moderation communities

### 4. Founder-led onboarding

For the first wave, sell manually:

- activate Pro manually with `/debug entitlements`
- onboard communities personally
- use supporter recognition for donors

This keeps the go-to-market simple while the product matures.

## Channel plan

### GitHub and docs

- Make the README clear on Free vs Pro
- Add screenshots of the Pro workflow
- Show `/debug entitlements` in owner docs

### Discord support server

- Keep one support server as the canonical demo and onboarding place
- Add channels for setup help, pricing questions, changelog, and showcase

### Reddit / community posts

Lead with the problem:

- support chaos
- no SLA visibility
- no incident workflow
- staff flying blind

Then position TON618 as the Discord-native solution.

### Short-form content

Create simple clips or GIFs for:

- opening a ticket
- showing `/config tickets`
- showing `/stats sla`
- using `/ticket playbook`
- enabling incident mode

## Offer structure

### Free

Position Free as fully usable, not crippled:

- tickets
- categories
- transcripts
- case context
- audit basics

### Pro

Position Pro as the operator upgrade:

- SLA rules
- auto-assignment
- incident mode
- daily reports
- live playbooks

### Supporter

Position Supporter as community backing:

- badge / recognition
- early access conversations
- no premium unlocks

## First 30 days

### Week 1

- Ship the English-first bot
- Update docs and screenshots
- Test Free and Pro flows in real servers

### Week 2

- Invite first design partners manually
- Activate Pro by owner command only
- Record setup friction and missing copy

### Week 3

- Publish the first public demos
- Post in 2-3 high-signal communities
- Collect testimonials and objections

### Week 4

- Refine pricing copy
- Improve onboarding docs
- Decide whether to automate billing in V2

## Messaging guardrails

- Do not position the bot as "just another ticket bot"
- Do not say donations unlock Pro
- Do not over-promise a full web billing system in V1
- Do say the product is bot-first and shipping commercially today

## Success metrics

- First 10 active servers
- First 3 Pro activations
- First 5 supporter guilds or donors
- Time-to-setup under 15 minutes for a new guild
- Clearer docs with fewer owner questions about pricing or activation
